I CPU

What is a CPU?

A central processing unit (CPU), also referred to as a processor, is the hardware within a computer that carries out the instructions of a computer program by performing the basic arithmetical, logical, and input/output operations of the system. CPU's are hardwired to support a specific set of operations, reffered to as it's instruction set architecture (ISA). Such operations may include adding or subtracting two numbers, comparing numbers, or jumping to a different part of a program. Each of these basic operations is represented by a particular sequence of bits; this sequence is called the opcode for that particular operation. Sending a particular opcode to a CPU will cause it to perform the operation represented by that opcode. To execute an instruction in a computer program, the CPU uses the opcode for that instruction as well as its arguments (for instance the two numbers to be multiplied, in the case of an multiplication operation). A computer program is therefore a sequence of instructions, with each instruction including an opcode and that operation's arguments. This program is generated by a compiler, which is aware of what ISA is being used by the processor. This is why you sometimes hear of C being described as platform dependent - any given computer may be using a different CPU and thus different ISA, meaning the program will have to be recompiled form source to run on that machine.

What architecture will be be talking about?

We will be talking about MIPS64, a reduced instruction set computer (RiSC) ISA, wherein every instruction/operation can be expressed in 64 bits, and every register (quick-access memory hardware that holds the results and operands of instructions) is 64-bits long as well. In order to process a sequence of these instructions, you can think of the processor going through the following five stages for each operation:

  • Instruction Fetch (IF) - send the program counter (PC) to memory and fetch the current instruction from memory.  Update the PC to the next sequential PC
  • Instruction Decode (ID) - Decode the instruciton (figure out which instruction from the ISA we want to process) and fetch the appropriate registers from the register file (specified by select bits in the instruction). Sign-extend the immediate field (a compiler-coded constant) of the instruction if the opcode indicates an immediate field is in use.  Deocding the instruction and fetching the registers at the same time is possible because the register specifiers are at a fixed location in a RISC architecture.
  • Execute (EX) - the Arithmetic Logic Unit (ALU) operates on the operands prepared in the prior cycle (a combination of registers fetch from register file and sign-extended immediate), performing one of three functions, depending on the instruction type. The three types of instructions are memory reference, register-register ALU instruction, register-immediate ALU instruction.
  • Memory Access (MEM) - If the instruction is a load, the control units reads data from memory using the effective address computed in the previous cycle. If it is a store, then the control unit writes the data from the specified register into memory at the effective address.
  • Write-back cycle (WB) - If a register was updated (e.g. by adding two other registers together and storing them in a third register, or loading from memory into a register), write the result into the register file, deciding whether it comes from the memory system (for a load) or from the ALU (for an ALU instruction) based on the operation being processed.

​So what does this have to do with pipelining?

Well, the processor's hardware is specialized, so that it has different pieces of hardware for different functions (e.g. the ALU is a different piece of hardware then than the register file and memory cache). Analogously, think of the different stages of an instruction being processed as the different stages in doing a load of laundry. In the first stage, it is one person's job to gather all of the clothes for the next load. The washing machine itself takes care of cleaning the clothes, and the dryer takes care of drying the clothes. Another person waits by the dryer to fold the clothes after they are dry. If we think of this "processor" as having 4 distinct pieces of hardware (where each person is a different person), than we want to use each piece of hardware efficiently. With any luck, every piece of hardware will be working at the same time. However, imagine we cannot start a load of laundry until the previous load of laundry is completely finished, folded and everything. At any moment in time, 3 pieces of hardware are being unused! This is what an unpipelined processor is like. Pipelining the process allows us to start gather the next load of laundry as soon as the previous one is put in the washing machine. Then, the washing machine can do it's job as the first person gathers another load. So far, each piece of "hardware" is performing as efficiently as they can. If this continues, then ideally a new load of laundry will be completely finished every cycle, where a cycle length is determined by the longest stage (i.e. even if the dryer only takes 5 minutes, and the washing machine takes 10, we can't move the clothes from the washing machine into the dryer until they are clean, so the dryer sits idle for 5 minutes). This is the benefit of pipelining!
