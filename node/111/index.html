GPU Architectures

The graphics pipeline and the computation of each of its sections are inherently parallel. The same calculations are made on a large set of data; The GPU was designed to accommodate these requirements. As a result, GPU cores use a single instruction, multiple data streams architecture.

 

To explain how a GPU core works, let’s start with a basic CPU style core.

In this diagram, this basic CPU style core runs the code to the right sequentially. This code shades a fragment. However, when computing various steps in the graphics pipeline, the same exact instructions are run except with a different set of data. A way to make this more efficient is to use  multiple cores that run simultaneously. Keep in mind that these cores do not have branch prediction or out of order execution logic. They are barebones cores.
 
 
This processor has 16 cores which all simultaneously run the same code. But, that’s a problem. If they all run the same instructions, instruction fetch and decode all produce the same results, and it results in unnecessary redundancy. To solve this problem, GPUs cores are designed to have multiple ALUs with a single instruction stream. In practice, a core usually has 16 to 64 ALUs each to process the respective number of fragments.
 

One problem that arises from this setup is the execution of branches. If all ALUs run the same instructions, a branch would break this rule. The solution to this problem is to execute both branches. Each ALU will go through all instructions but stall when executing instructions that do not occur in the branch that it has taken.

All of this is nice, but there is still a major problem. CPUs can quickly run code sequentially in part to the caches that they have. But in the case of GPUs, each core cannot have large caches due to costs. Remember that the cores are stripped down versions of fancy CPUs. As a result, each core can experience a considerable amount of stalling when waiting for memory. To solve this, a GPU core executes on multiple sets of fragments. When one stalls, it switches to another set and so on. It operates much like a computer multitasks; It switches between execution contexts.
