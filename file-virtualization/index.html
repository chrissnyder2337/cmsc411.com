File Virtualization

The Basics

File virtualization is used to eliminate the dependencies between data accessed at the file level and the location where the files are physically stored. This creates a system that is analogous to the DNS lookup system, users can access data through the virtualization without knowing the exact address just as they can access a website through a DNS server without knowing the exact IP address of the server the site is hosted on. In its simplest form, file virtualization works almost the exact same way, where the virtualization serves as a sort of "lookup", mapping file names to their physical locations which may or may not be hidden to the end user.

The Benefits

This form of virtualization is most beneficial to large companies that have vast amounts of data hosted on various servers in different locations, that needs to be accessible at all times by its clients or employees. It enables the storage manager to freely move data around without informing every user of the new location. At the most basic level, this is good as it hides unecessary information from the end user, leaving a simplistic and minimalist form of accessing data, requiring the user to only remember one keyword or name rather than the new address on the storage system every time it is moved or updated, reducing the potential for human error.

The largest improvement, however, is in allowing multi-tiered storage networks. These networks, as the name implies, have multiple tiers of servers, which prioritize various aspects of storage systems. Generally, the lower tiers are more power-efficient machines, which cost less to run and maintain while also providing lower speeds, while the higher tiers are more speed oriented, while costing more to the company. The reason file virtualization aids in this type of network, is that by being able to freely move data to new locations without informing the user, the storage manager can move less frequently accessed items to the lower tier, while prioritizing frequently accessed data in the top tier. This saves the company money through its increased efficiency.

"Global" File System

There are two methods to file virtualization. The first is a "global" file system. In this method, the virtualization is built straight into the storage system intself. This means that the file system manages the metadata that contains the mapped file locations, which allows for quicker lookups, but at a major cost: expansion of the system must use the same hardware. This obviously limits the flexibility to mix hardware in a system, as the hardware generally needs to be made by the same manufacturer, as well as usually either the same or similar model. By not being able to mix hardware, the system cannot take advantage of a multi-tiered system as previously described, which in turn leads to higher costs. This sytem still, however, provides the other benefits of file virtualization, and is generally not granular to the file level, requiring entire folders to meet various conditions before being migrated to another location.

Stand Alone Virtualization Engine

The second method is the use of a stand alone virtualization engine. These engines are separate from the storage system itself, and instead serve as a middleman between the user and the storage system. The major advantage of this method over the previous is the flexibility of the hardware used in the storage system itself. Since the virtualization happens separate from the storage system, it is possible for the system to be heterogenous, which allows ease of expansion, as well as the ability to use a multi-tiered system, the benefits of which can be found in the previous sections.

There are two methods to implementing a stand alone virtualization engine, in-band and out-of-band. In-band means that the virtualization engine is in the data path, or that all I/O requests and their data go through the device. The user never interacts with the storage system itself, instead only dealing with the middleman. This method allows for caching data, data usage statistics, replications services, and data migration to all be easily implemented, as the data actually passes through the virtualization device.

Out-of-band means the device is out of the data path. In this method, I/O requests to the storage system are intercepted before actually leaving, and a lookup is sent to the meta-data server, which returns the physical location of the data. The information is then retrieved through an actual I/O request to the storage using the recently found location. Caching is not possible in this setup, as the data never actually passes through the lookup server.
